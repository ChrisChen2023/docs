Every request you make to the GPU Cloud API must be authenticated with an API key. API keys act as long-lived credentials that identify both your account and the level of access granted to the caller. You generate and manage keys entirely within the GPU Cloud console — no OAuth flows or token exchanges are required. Keep your keys safe: anyone who holds a key can take actions on your behalf up to that key's scope.

## Generating an API Key

1. Log in to the [GPU Cloud Console](https://console.gpucloud.com).
2. Navigate to **Settings → API Keys**.
3. Click **Create API Key**.
4. Give the key a descriptive name (e.g., `ci-pipeline-prod`) and select the appropriate scope (see [Key Scopes](#key-scopes) below).
5. Click **Generate**. Copy the key immediately — it is only shown once in full at creation time.

<Note>
  After you close the creation dialog, GPU Cloud only displays the last four characters of each key. Store the full key in a secure secrets manager as soon as you generate it.
</Note>

## Passing Your API Key

Include your API key as a Bearer token in the `Authorization` header of every request:

```http
Authorization: Bearer <your-api-key>
```

The API does not support key-in-URL query parameters or cookie-based authentication. Always transmit your key over HTTPS — the API rejects plain HTTP connections.

### Examples

<CodeGroup>

```bash curl
curl -X GET https://api.gpucloud.com/v1/instances \
  -H "Authorization: Bearer <your-api-key>" \
  -H "Accept: application/json"
```

```python Python
import os
import requests

api_key = os.environ["GPUCLOUD_API_KEY"]

response = requests.get(
    "https://api.gpucloud.com/v1/instances",
    headers={
        "Authorization": f"Bearer {api_key}",
        "Accept": "application/json",
    },
)

response.raise_for_status()
print(response.json())
```

```javascript JavaScript
const apiKey = process.env.GPUCLOUD_API_KEY;

const response = await fetch("https://api.gpucloud.com/v1/instances", {
  method: "GET",
  headers: {
    Authorization: `Bearer ${apiKey}`,
    Accept: "application/json",
  },
});

if (!response.ok) {
  const error = await response.json();
  throw new Error(`API error: ${error.error.code} — ${error.error.message}`);
}

const instances = await response.json();
console.log(instances);
```

</CodeGroup>

<Tip>
  Store your API key in an environment variable (e.g., `GPUCLOUD_API_KEY`) rather than hard-coding it in your source files. All three examples above read the key from the environment — adopt this pattern from day one to keep credentials out of your codebase and CI logs.
</Tip>

## Key Scopes

When you create an API key you choose one of two scopes. Pick the most restrictive scope that meets your use case — this limits the blast radius if a key is ever leaked.

| Scope | Description |
|---|---|
| **Read-Only** | Allows `GET` requests only. Suitable for monitoring dashboards, read-only CI checks, and audit scripts. Cannot create, modify, or delete any resources. |
| **Full Access** | Allows all request methods (`GET`, `POST`, `PATCH`, `DELETE`). Required for provisioning instances, managing snapshots, and any write operation. |

<Note>
  You can create multiple API keys with different scopes and names. Use separate keys for separate systems (e.g., one for your CI pipeline, one for your deployment scripts) so you can rotate or revoke them independently.
</Note>

## If a Key Is Compromised

If you suspect an API key has been exposed — through a leaked repository, log file, or any other channel — revoke it immediately:

1. Go to **Settings → API Keys** in the [GPU Cloud Console](https://console.gpucloud.com).
2. Locate the compromised key by name or suffix.
3. Click **Revoke**. Revocation is instantaneous; any in-flight request using that key will receive a `401 unauthorized` error.
4. Generate a replacement key and update your secrets store and any services that relied on the old key.

<Warning>
  Never commit API keys to source control — not even in private repositories. Secrets committed to Git persist in history even after deletion and can be extracted by anyone with repository access. Use a secrets manager (such as AWS Secrets Manager, HashiCorp Vault, or your CI platform's built-in secrets store) to inject credentials at runtime.
</Warning>

## Authentication Errors

If your key is missing or invalid, the API returns a `401 unauthorized` response:

```json
{
  "error": {
    "code": "unauthorized",
    "message": "Invalid or missing API key. Provide a valid Bearer token in the Authorization header."
  }
}
```

Common reasons for a `401` error:

- The `Authorization` header is absent from the request.
- The key has been revoked in the console.
- The key value was copied with surrounding whitespace or truncated.
- You are targeting the wrong environment (e.g., using a production key against a staging endpoint).
