# Mintlify Agent Conventions

This file defines the strict instructions and conventions for the Mintlify Agent when modifying or maintaining our documentation.

## General Documentation Rules
- Always structure content using the Diátaxis framework.
- Ensure every newly created MDX page contains standard frontmatter with `title`, `description`, and `sidebarTitle`.
- Use the built-in standard Mintlify components (like `<CardGroup>` and `<Accordion>`) instead of custom HTML tags whenever possible.

## API Documentation Conventions
- When updates are proposed based on OpenAPI schema changes, strictly sync fields with the `openapi.json` properties.
- Parameter descriptions must use clear, gender-neutral technical language.
- Every endpoint must document: authentication requirements, rate limits, and common error codes.
- Use real-world parameter values in examples (not foo/bar placeholders).
- Include a complete request/response cycle for each endpoint.

## Pull Request Guidelines
- All PRs generated by the agent must begin with the branch prefix `docs/ai-update/`.
- Every PR description must include a "Changes Made" section listing all altered file paths, and a "Reason" section explaining why the change was triggered.
- Automatically assign @TechWriterLead as the primary reviewer.
- Tag the PR with the label `mintlify-agent`.

## Style and formatting
- Write for developers with 2-5 years of experience. Don't oversimplify, but explain non-obvious concepts.
- Use active voice and second person ("you").
- Date format: ISO 8601 (YYYY-MM-DD).
- When referencing UI elements, use bold: **Settings** button.


## What to include
- Add prerequisite sections to guides when users need API keys, environment setup, or dependencies.
- Include "Next steps" sections linking to related documentation.
- Add troubleshooting sections for common issues we see in support tickets.
