A GPU instance is a virtual machine backed by dedicated GPU hardware that you provision on demand through GPU Cloud. Each instance combines a specific GPU model, vCPU allocation, system memory, and local NVMe storage into a single compute unit you can launch in seconds. Whether you are training a large language model, running batch inference, or exploring a new framework, instances give you full root access to isolated, reproducible GPU environments — without the overhead of managing physical hardware.

## Instance Lifecycle

Every instance on GPU Cloud moves through a well-defined set of states from the moment you request it to the moment it is removed. Understanding these states helps you write reliable automation, interpret billing accurately, and avoid accidental data loss.

<Steps>
  <Step title="Pending">
    Your instance has been accepted and GPU Cloud is allocating hardware, loading your chosen disk image, and injecting your SSH key. No charges accrue during this phase.
  </Step>
  <Step title="Running">
    The instance is live and fully accessible. vCPUs, GPU, memory, and storage are all active. Billing begins the moment the instance enters this state.
  </Step>
  <Step title="Stopped">
    The instance OS has been shut down but the underlying hardware allocation is retained. Your ephemeral disk is preserved. Billing continues at a reduced rate because the GPU reservation is still held on your behalf.
  </Step>
  <Step title="Terminated">
    The instance has been permanently destroyed. All ephemeral (instance) storage is deleted and the hardware is released back to the pool. This action is irreversible — attach a persistent volume before terminating if you need to keep your data.
  </Step>
</Steps>

<Warning>
  Terminating an instance permanently deletes all data stored on its local NVMe disk. Snapshot or migrate any work you want to keep before you terminate.
</Warning>

## Instance States Reference

The table below summarises every state you may encounter in the dashboard, API, or CLI.

| State | Description | Billable | Accessible via SSH |
|---|---|---|---|
| `pending` | Instance is being provisioned | No | No |
| `running` | Instance is fully operational | Yes — full rate | Yes |
| `stopping` | OS shutdown in progress | Yes — full rate | No |
| `stopped` | Instance is halted; hardware reserved | Yes — reduced rate | No |
| `starting` | Instance is resuming from stopped | Yes — full rate | No |
| `terminating` | Instance is being destroyed | No | No |
| `terminated` | Instance has been permanently removed | No | No |
| `error` | Provisioning or runtime failure occurred | No | No |

## Key Instance Properties

Every instance has a set of immutable and mutable properties you can inspect at any time via the dashboard or API.

<CardGroup cols={2}>
  <Card title="Instance ID" icon="fingerprint">
    A unique identifier (e.g., `inst_a1b2c3d4`) assigned at creation. Use this in API calls and CLI commands to target a specific instance.
  </Card>
  <Card title="GPU Type" icon="microchip">
    The GPU model attached to the instance (e.g., `NVIDIA H100 80GB SXM5`). Set at launch and cannot be changed without re-provisioning.
  </Card>
  <Card title="Region" icon="globe">
    The data-center location where your instance runs (e.g., `us-east-1`). Choosing a region close to your data reduces transfer latency and costs.
  </Card>
  <Card title="IP Address" icon="network-wired">
    A public IPv4 address auto-assigned when the instance starts. You can optionally reserve a static IP to keep the same address across restarts.
  </Card>
  <Card title="Status" icon="circle-info">
    The current lifecycle state of the instance. Poll this field to build launch-and-wait automation or monitor long-running jobs.
  </Card>
  <Card title="Disk Image" icon="hard-drive">
    The base operating system and pre-installed software stack (e.g., `pytorch-2.3-cuda12`). Selected at launch and baked into the boot volume.
  </Card>
  <Card title="SSH Key" icon="key">
    The public key injected into the instance at provisioning time. You must have the corresponding private key to connect.
  </Card>
  <Card title="Instance Type" icon="server">
    The hardware profile that determines the number of GPUs, vCPUs, and RAM allocated to your instance (e.g., `gpu.h100.1x`).
  </Card>
</CardGroup>

## Interacting with Instances

You can manage instances through three equally capable interfaces — choose whichever fits your workflow.

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Instances** in the left sidebar, then click **Launch Instance** to open the configuration wizard. You can start, stop, and terminate instances from the instance detail page.
  </Tab>
  <Tab title="CLI">
    ```bash
    # Launch a new instance
    gpucloud instances launch \
      --gpu h100-80gb-sxm5 \
      --image pytorch-2.3-cuda12 \
      --region us-east-1 \
      --ssh-key my-key

    # List running instances
    gpucloud instances list --status running

    # Stop an instance
    gpucloud instances stop inst_a1b2c3d4

    # Terminate an instance
    gpucloud instances terminate inst_a1b2c3d4
    ```
  </Tab>
  <Tab title="API">
    ```bash
    # Launch a new instance
    curl -X POST https://api.gpucloud.com/v1/instances \
      -H "Authorization: Bearer $GPUCLOUD_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "gpu_type": "h100-80gb-sxm5",
        "image": "pytorch-2.3-cuda12",
        "region": "us-east-1",
        "ssh_key_name": "my-key"
      }'

    # Get instance details
    curl https://api.gpucloud.com/v1/instances/inst_a1b2c3d4 \
      -H "Authorization: Bearer $GPUCLOUD_API_KEY"
    ```
  </Tab>
</Tabs>

<Tip>
  Use the `--wait` flag with the CLI (`gpucloud instances launch --wait`) to block until your instance reaches the `running` state — ideal for scripted pipelines that need to SSH in immediately after launch.
</Tip>
